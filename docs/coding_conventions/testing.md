# コーディング規約: テスト(pytest / Vitest)

> このファイルは`backend/tests`・`frontend/src`の実装済みテストの実態に基づいて記載する。

## 1. 命名規則

- backend: ファイルは`test_<対象領域>.py`(例: `test_auth.py`, `test_order_cancel_return.py`)。テスト関数は`test_<操作>_<期待結果>`の形式にする(例: `test_login_wrong_password_returns_401`)。関数名だけで「何をして」「何を期待するか」がわかることを目指す
- frontend: ファイルは`<対象コンポーネント>.test.jsx`とし、テスト対象と同じディレクトリに置く(例: `pages/CartView.test.jsx`)

## 2. テスト構造

- AAA(Arrange-Act-Assert)を基本構造とする。準備・実行・検証を明確に分け、1テストケースにつき検証したい振る舞いを1つに絞る
- フロントエンドのテストは実装の内部詳細(state変数名など)ではなく、ユーザーから見た振る舞い(表示されるテキスト、クリック操作の結果)を検証する

## 3. フィクスチャ・モック

- backend: 共通フィクスチャは`conftest.py`に集約する(`client`, `auth_headers`, `admin_headers`等)。DBは`_reset_db`(autouse fixture)で各テスト前にリセットし、テスト間の状態共有を避ける
- 外部サービス(Stripe等)は`monkeypatch`でモックする(`mock_stripe_refund_success`等の既存パターンに合わせる)。モックする境界は「外部API呼び出し」に限定し、アプリ内部のロジック(ルーター・DB操作)はモックせず実際に実行する
- frontendのAPI呼び出しは`vi.mock`等で`api/`層をモックし、コンポーネントの振る舞いを検証する

## 4. テストの独立性

- 各テストは実行順序に依存せず、単独で実行しても他のテストと同じ結果になるようにする
- グローバルな状態(モジュールレベル変数等)をテスト間で共有しない。DBは`_reset_db`のようなautouseフィクスチャでリセットする

## 5. カバレッジ運用

- backend: `pytest-cov`によるカバレッジゲートを70%に設定済み(`pyproject.toml`)。この数値は「最低限」であり、目標ではない。100%を無理に目指して意味の薄いテスト(アサーションのないテスト等)を追加しない
- frontend: Vitestのカバレッジ閾値ゲートも同様の考え方とする
- 新機能を追加する際は、正常系だけでなく異常系(バリデーションエラー、権限エラー、404等)もテストに含める(既存の`test_*_returns_4xx`系のテストパターンに合わせる)
- 決済(Stripe)・注文・認可まわりは、他の領域よりテスト密度を優先的に高める
