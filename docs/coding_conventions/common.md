# コーディング規約: 共通(言語非依存・プロジェクト運営)

## 1. 命名規則

- Pythonファイル/変数/関数は `snake_case`、クラスは `PascalCase`、定数は `UPPER_SNAKE_CASE`(例: `backend/app/auth.py` の `SECRET_KEY`, `ALGORITHM`)
- JS/Reactのファイル/変数/関数は `camelCase`、コンポーネントファイル・コンポーネント名は `PascalCase`(例: `ProductDetail.jsx`)
- 同じ概念(例: 商品ID)を指す変数名は言語をまたいでも意味を統一する(`product_id` ⇔ `productId`)
- 1文字変数やi, jのような一時変数はループカウンタ等の極めて限定的な場面のみ許容し、それ以外は意味のある名前を付ける

## 2. Git / コミット運用

- コミットメッセージは [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) 形式 `<type>: <summary>` を用いる
  - 主な `type`: `feat`(新機能) / `fix`(バグ修正) / `docs`(ドキュメントのみ) / `refactor`(挙動を変えない構造変更) / `test`(テスト追加・修正) / `chore`(依存更新・雑務)
  - 本プロジェクトはscopeまでは必須にしない(個人開発のため厳密な運用コストに見合わない)。例: `feat: 低在庫アラート機能を追加`, `fix: passlibをbcrypt直接呼び出しに置き換え`
- 1コミット = 1つの論理的な変更に留める(機能追加とリファクタを同一コミットに混ぜない)
- 本プロジェクトは個人開発のため、`main`ブランチへの直接コミット・pushを正式なワークフローとする(PRベースの運用は採用しない)

## 3. コメント

- コメントは「なぜ(WHY)」を書く。「何を(WHAT)」しているかは、良い名前の変数・関数で自己説明させる
- 非直感的な実装・ワークアラウンド(例: 特定バージョンの依存ライブラリの制約を回避するコード)には、理由をコメントで残す
- コードを変更する際、関連するコメントが古くなっていないか確認し、実態と合わなくなったコメントは更新または削除する
- 多段落のコメントブロックや冗長なdocstringは書かない(このプロジェクトのCLAUDE.md方針に準拠)

## 4. ドキュメンテーション

- リポジトリ直下の`README.md`(存在する場合)には最低限、プロジェクト概要・セットアップ手順・起動方法を記載する
- 設計・要件レベルのドキュメントは`docs/deliverables/`配下のドキュメント駆動フロー(`docs/README.md`参照)に従う。本規約(`docs/coding_conventions/`)はそのフローとは別の「実装時の書き方」を扱う

## 5. セキュリティ

- シークレット(APIキー、`SECRET_KEY`、Stripeキー等)はコードにハードコードせず環境変数から読む(`os.environ.get(...)`)。`.env`はgit管理対象外とする
- ユーザー入力はAPI境界(Pydanticスキーマ / リクエストバリデーション)で必ず検証する。SQLインジェクション対策としてSQLAlchemyのORM/パラメータバインディングを経由しない生SQL文字列結合は行わない
- 依存パッケージ(`uv.lock` / `package-lock.json`)はDependabotによる週次更新を受け入れ、`pip-audit`(backend)で脆弱性がないか定期的に確認する

## 6. エラーハンドリング / ログ

- ユーザー向けエラーメッセージ(`HTTPException.detail`等)は日本語で簡潔に、原因を特定されすぎない範囲で記述する(例: 「認証情報が無効です」であって、DBのカラム名やスタックトレースを含めない)
- ログ(`logging_config.py`経由)にパスワード・トークン・カード情報等の機密情報を出力しない
- 開発環境と本番環境でログレベル・デバッグ情報の出力範囲を分ける(FastAPIの`debug`モードは本番で無効にする)
