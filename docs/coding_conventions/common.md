# コーディング規約: 共通(言語非依存・プロジェクト運営)

## 0. 本規約の位置づけ

本規約は既存実装の説明書ではなく、今後の実装とレビューで守る基準である。ルールを決める際は、次の優先順位で根拠を確認する。

1. 言語・フレームワーク・ライブラリの公式仕様および公式ドキュメント
2. OWASP、WCAG等の公開標準・業界ガイドライン
3. 本プロジェクトの要件・設計上の制約
4. 既存実装

既存実装が上位の根拠と異なる場合、既存実装を理由に標準を引き下げず、次のいずれかに分類する。

- **標準**: 新規・変更コードが従うルール。本文中の「〜する」「〜しない」は必須を表す
- **例外**: 標準から意図的に外す判断。理由・影響範囲・見直し条件を本文または設計書に残す
- **移行対象**: 既存コードには残っているが、新規コードでは増やさず、関連箇所を変更する際に段階的に解消するもの
- **要決定**: 複数の妥当な選択肢があり、要件または運用条件を確認してから決めるもの

規約と実装に差異を見つけた場合は、規約を無条件に実装へ合わせない。規約の根拠が妥当なら実装を移行対象とし、根拠が変わった場合のみ規約を改訂する。例外を恒久化せず、依存関係の主要更新やアーキテクチャ変更時に見直す。

## 1. 命名規則

- Pythonファイル/変数/関数は `snake_case`、クラスは `PascalCase`、定数は `UPPER_SNAKE_CASE`(例: `backend/app/auth.py` の `SECRET_KEY`, `ALGORITHM`)
- JS/Reactのファイル/変数/関数は `camelCase`、コンポーネントファイル・コンポーネント名は `PascalCase`(例: `ProductDetail.jsx`)
- 同じ概念(例: 商品ID)を指す変数名は言語をまたいでも意味を統一する(`product_id` ⇔ `productId`)
- 1文字変数やi, jのような一時変数はループカウンタ等の極めて限定的な場面のみ許容し、それ以外は意味のある名前を付ける

## 2. Git / コミット運用

- コミットメッセージは [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) 形式 `<type>: <summary>` を用いる
  - 主な `type`: `feat`(新機能) / `fix`(バグ修正) / `docs`(ドキュメントのみ) / `refactor`(挙動を変えない構造変更) / `test`(テスト追加・修正) / `chore`(依存更新・雑務)
  - 本プロジェクトはscopeまでは必須にしない(個人開発のため厳密な運用コストに見合わない)。例: `feat: 低在庫アラート機能を追加`, `fix: passlibをbcrypt直接呼び出しに置き換え`
- 1コミット = 1つの論理的な変更に留める(機能追加とリファクタを同一コミットに混ぜない)
- 現在は個人学習プロジェクトとして、`main`ブランチへの直接コミット・pushを**プロジェクト固有の例外**として許容する。push後はCI結果を確認し、失敗した状態を放置しない
- 共同開発への移行、外部公開、本番運用のいずれかを開始する場合は、ブランチ保護・必須ステータスチェック・PRレビューを導入し、この例外を見直す

## 3. コメント

- コメントは「なぜ(WHY)」を書く。「何を(WHAT)」しているかは、良い名前の変数・関数で自己説明させる
- 非直感的な実装・ワークアラウンド(例: 特定バージョンの依存ライブラリの制約を回避するコード)には、理由をコメントで残す
- コードを変更する際、関連するコメントが古くなっていないか確認し、実態と合わなくなったコメントは更新または削除する
- 多段落のコメントブロックや、型・関数名から明らかな内容を繰り返すdocstringは書かない。公開APIの契約や非自明な制約の説明に必要な長さは許容する

## 4. ドキュメンテーション

- リポジトリ直下の`README.md`には最低限、プロジェクト概要・前提条件・セットアップ手順・起動方法・テスト方法を記載する
- 設計・要件レベルのドキュメントは`docs/deliverables/`配下のドキュメント駆動フロー(`docs/README.md`参照)に従う。本規約(`docs/coding_conventions/`)はそのフローとは別の「実装時の書き方」を扱う

## 5. セキュリティ

- シークレット(APIキー、`SECRET_KEY`、Stripeキー等)はコードにハードコードしない。ローカル開発ではgit管理外の`.env`等、本番では実行環境のシークレット管理機構から注入する。環境変数を使う場合も値をログへ出力しない
- セキュリティ上必須の設定値は、本番環境で未設定の場合に安全でないデフォルト値へフォールバックせず、起動時に失敗させる(fail closed)
- ユーザー入力はAPI境界(Pydanticスキーマ / リクエストバリデーション)で必ず検証する。SQLインジェクション対策としてSQLAlchemyのORM/パラメータバインディングを経由しない生SQL文字列結合は行わない
- 依存パッケージ(`uv.lock` / `package-lock.json`)はロックファイルを更新して再現性を保つ。Dependabotによる更新は、テスト・互換性・脆弱性情報を確認して取り込む
- backendは`pip-audit`、frontendはDependabot alert等の利用可能な監査手段で脆弱性を確認する。脆弱性を一時的に除外する場合は、理由・影響・修正版の有無・見直し条件をCI設定に残す

## 6. エラーハンドリング / ログ

- ユーザー向けエラーメッセージ(`HTTPException.detail`等)は日本語で簡潔に、原因を特定されすぎない範囲で記述する(例: 「認証情報が無効です」であって、DBのカラム名やスタックトレースを含めない)
- ログ(`logging_config.py`経由)にパスワード・トークン・カード情報等の機密情報を出力しない
- 開発環境と本番環境でログレベル・デバッグ情報の出力範囲を分ける(FastAPIの`debug`モードは本番で無効にする)
- 例外を握りつぶさない。回復できない例外は、機密情報を除いた診断可能なコンテキストとともに記録し、呼び出し元へ失敗を伝播させる
- ログには必要に応じてリクエストID、対象リソースID、処理結果を含める。ただし個人情報や認証情報を識別子代わりに使わない
