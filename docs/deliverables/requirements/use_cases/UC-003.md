# UC-003: カード決済を使わずに注文を確定する

[← ユースケース一覧に戻る](../01_use_cases.md)

**業務領域**: 商品購入業務

**注記**: `frontend/src/pages/CartView.jsx`(`CartView`)を確認した結果、カート画面には「注文を確定する」ボタンが常時表示されており、Stripeを使わずに即時注文確定する経路(`POST /orders`)が実装として存在することが判明した。要件定義フェーズの完成時点でこの経路の記載が漏れていたため、本ユースケースを追加した

**元になったUser Story**: US-002, US-003(カート・注文金額確認に関する既存のUser Storyに準ずる。カード決済を伴わない注文確定を対象とする)
**主アクター**: 顧客
**ステークホルダーと関心事**:
- 顧客: カード情報を入力せずに、代金引換・銀行振込等の想定で注文を確定させたい
- 運営: 在庫・クーポン使用回数の整合性を、決済有無に関わらず保ちたい

**事前条件**: カートに商品が1つ以上入っている(Stripe決済の有効/無効は問わない)
**事後条件(成功時)**: 注文(status=pending)が作成され、在庫が減算され、カートが空になり、確認メールが送信されている

**基本フロー**:
1. 顧客が、カート画面で「注文を確定する」ボタンを押す
2. システムが、カートに商品が1つ以上あることを確認する
3. システムが、カート内商品それぞれについて在庫数(`stock`)が数量以上であることを確認する
4. システムが、適用中クーポンがあれば有効性(有効フラグ・使用回数上限)を確認し、割引額を計算する
5. システムが、小計・割引額・消費税(10%)から合計金額を計算する
6. システムが、注文(status=pending)を作成し、注文明細を登録する
7. システムが、適用されたクーポンの使用回数を1増やす
8. システムが、購入された商品の在庫数を減算する
9. システムが、カート内商品を削除する
10. システムが、注文確認メールを送信する

**代替フロー**: なし

**例外フロー**:
- 2a. カートが空の場合
  1. システムが、「カートが空です」エラーを表示する
- 3a. カート内商品のいずれかで在庫が不足している場合
  1. システムが、「{商品名}の在庫数が不足しています」エラーを表示する
- 4a. 指定されたクーポンコードが無効な場合
  1. システムが、「無効なクーポンコードです」エラーを表示する
- 4b. クーポンの使用回数が上限に達している場合
  1. システムが、「このクーポンは使用回数の上限に達しています」エラーを表示する

**関連する概念エンティティ**: 顧客, カート, 注文, 注文明細, クーポン, 商品
**関連する画面**: カート画面
