# UC-008: 返品を承認・却下する(管理者)

[← ユースケース一覧に戻る](../01_use_cases.md)

**業務領域**: 注文管理業務(管理者向け)

**元になったUser Story**: US-023
**主アクター**: 管理者
**ステークホルダーと関心事**:
- 管理者: 現物の返送状況等を踏まえて、適切な申請のみ承認し返金・在庫戻しを行いたい
- 顧客: 申請の結果(承認/却下)を速やかに知りたい

**事前条件**: 管理者(`is_admin=true`)がログイン済みであり、対象の注文のステータスが`return_requested`である
**事後条件(成功時・承認)**: 対象注文のステータスが`returned`になる。注文明細の数量分だけ商品在庫が加算される。クーポンを使用していた場合は使用回数が1減算される。Stripe決済済みの場合はStripe側で返金が実行される。顧客に通知メールが送信される
**事後条件(成功時・却下)**: 対象注文のステータスが`shipped`に戻る。在庫・クーポン使用回数・返金は変更されない。顧客に通知メールが送信される

**基本フロー**:
1. 管理者が、注文管理画面(S-103)で返品申請中の注文一覧を確認する
2. 管理者が、対象の注文について「承認」または「却下」を選択する
3. システムが、対象注文が存在し、ステータスが`return_requested`であることを確認する
4. (承認の場合) システムが、対象注文に`stripe_payment_intent_id`が設定されている場合、Stripe返金API(`stripe.Refund.create`)を呼び出す
5. (承認の場合) システムが、注文明細の数量分だけ商品在庫を加算し、適用されていたクーポンの`used_count`を1減算し、注文ステータスを`returned`に更新する
6. (却下の場合) システムが、注文ステータスを`shipped`に更新する(在庫・クーポン使用回数・返金は変更しない)
7. システムが、ステータス変更通知メールを送信する(N-002拡張。送信失敗時もフロー自体は継続する)

**代替フロー**: なし

**例外フロー**:
- 3a. 対象注文が存在しない場合
  1. システムが、「注文が見つかりません」エラーを表示する(404)
- 3b. 対象注文のステータスが`return_requested`以外の場合
  1. システムが、「この注文は返品申請中ではありません」エラーを表示する(400)
- 4a. Stripe返金APIの呼び出しが失敗した場合
  1. システムが、「返金処理に失敗しました」エラーを表示する(500)。注文ステータス・在庫・クーポン使用回数は変更しない

**関連する概念エンティティ**: 顧客(CUSTOMER)、注文(ORDER)、注文明細(ORDER_ITEM)、商品(PRODUCT)、クーポン(COUPON)
**関連する画面**: 注文管理画面(S-103)

**備考**:
- 却下時に在庫・クーポン使用回数を戻さない理由: 現物は顧客の手元にあるままであり、返送された事実が確認できていないため(UC-006のキャンセルとは異なり、返品は現物の受領確認を伴う業務のため管理者判断を挟む)
- 承認・却下は管理者の一度きりの判断とし、判断後の取り消し(再承認・再却下)は本ユースケースのスコープ外とする
