# UC-009: パスワードをリセットする

[← ユースケース一覧に戻る](../01_use_cases.md)

**業務領域**: 会員管理業務

**元になったUser Story**: US-026
**主アクター**: 顧客(未ログイン、パスワードを忘れた状態)
**ステークホルダーと関心事**:
- 顧客: パスワードを忘れても自力でログインを回復したい。他人に自分のメールアドレスの登録有無を知られたくない
- 運営: リセット機能が第三者による不正アクセス(他人のアカウント乗っ取り)の踏み台にならないようにしたい。登録済みメールアドレスの列挙(ユーザー列挙攻撃)を許さないようにしたい

**事前条件**: 顧客がログイン画面(S-005)にいる。パスワードを忘れてログインできない状態である
**事後条件(成功時)**: 対象アカウントのパスワードが新しいハッシュ値に更新される。使用したリセットトークンは無効化され、再利用できなくなる

**基本フロー**:
1. 顧客が、ログイン画面(S-005)で「パスワードをお忘れですか?」を選択する
2. システムが、パスワードリセット要求画面(S-008)を表示し、メールアドレスの入力を求める
3. 顧客が、メールアドレスを入力し、確定する
4. システムが、入力されたメールアドレスに一致する有効なアカウント(`is_active=true`)が存在するか確認する
5. システムが、存在する場合はランダムなリセットトークンを生成し、有効期限(発行から24時間)とともに保存したうえで、リセットリンク(`{FRONTEND_URL}/reset-password?token=...`)付きのメールを送信する(N-005)
6. システムが、アカウントの存在有無に関わらず同一の完了画面(「メールを送信しました」)を表示する
7. 顧客が、受信したメール内のリセットリンクを開き、パスワード再設定画面(S-009)で新しいパスワードを入力し、確定する
8. システムが、トークンが有効(存在し、未使用かつ期限内)であることを確認する
9. システムが、新しいパスワードをハッシュ化して保存し、リセットトークンを無効化する(再利用不可にする)
10. システムが、「パスワードを更新しました」を表示し、ログイン画面(S-005)に遷移する

**代替フロー**: なし

**例外フロー**:
- 4a. 入力されたメールアドレスに一致する有効なアカウントが存在しない場合
  1. システムは、メールを送信せず、基本フロー6と同一の完了画面を表示する(顧客からはメール送信の成否を区別できない)
- 8a. トークンが存在しない、既に使用済み、または発行から24時間を超えて期限切れの場合
  1. システムが、「リンクが無効です。再度お手続きください」エラーを表示する(400)。パスワードは更新されない

**関連する概念エンティティ**: 顧客(CUSTOMER)
**関連する画面**: ログイン画面(S-005)、パスワードリセット要求画面(S-008)、パスワード再設定画面(S-009)

**備考**:
- メールアドレスの存在有無を顧客に開示しない設計(例外フロー4a)は、ユーザー列挙攻撃を防ぐための意図的なセキュリティ対策であり、実装漏れではない
- リセットトークンはJWTではなく、DBに保存するランダムな不透明トークン(1回使用で失効)を用いる。JWTはステートレスで個別失効の仕組みを持たないため、パスワードリセットのような「1回限りの使い捨てトークン」の用途には不向きと判断した
- パスワードリセット後も、リセット以前に発行済みのログインセッション(JWTアクセストークン)は個別には失効しない(本システムのJWTにはトークン単位の失効の仕組みがないため)。この制約は将来的な課題として扱う
