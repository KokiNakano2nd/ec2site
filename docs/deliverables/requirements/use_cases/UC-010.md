# UC-010: メールアドレスの本人確認を行う

[← ユースケース一覧に戻る](../01_use_cases.md)

**業務領域**: 会員管理業務

**元になったUser Story**: US-027
**主アクター**: 顧客
**ステークホルダーと関心事**:
- 顧客: 誤って他人のメールアドレスを登録してしまっていないか確認したい。確認作業が購入等の他の操作の妨げにならないようにしたい
- 運営: 実在しないメールアドレスでのアカウント作成・悪用を防ぎたい。ただし確認を必須(ブロッキング)にすることで新規顧客の離脱を招きたくない

**事前条件**: 顧客が会員登録を完了している
**事後条件(成功時)**: 対象アカウントの`is_verified`が`true`に更新される。使用した確認トークンは無効化され、再利用できなくなる

**基本フロー**:
1. 顧客が、ログイン画面(S-005)で会員登録する(既存フロー、US-006/US-007)
2. システムが、アカウントを作成する(`is_verified=false`)と同時に、ランダムな確認用トークンを生成し、有効期限(発行から7日間)とともに保存する
3. システムが、登録されたメールアドレス宛に確認リンク(`{FRONTEND_URL}/?verify_token=...`)付きのメールを送信する(N-006)
4. 顧客が、受信したメール内の確認リンクを開く
5. システムが、トークンが有効(存在し、未使用かつ期限内)であることを確認する
6. システムが、対象アカウントの`is_verified`を`true`に更新し、確認トークンを無効化する(再利用不可にする)
7. システムが、「メールアドレスを確認しました」を表示する

**代替フロー**:
- 3a. 顧客が確認メールを受信できなかった、または見つけられなかった場合
  1. 顧客が、ログイン後にプロフィール画面(S-007)で「確認メールを再送する」を選択する
  2. システムが、新しい確認用トークンを生成し(既存のトークンを上書き)、確認リンク付きのメールを再送する(基本フロー3に合流)

**例外フロー**:
- 5a. トークンが存在しない、既に使用済み、または発行から7日を超えて期限切れの場合
  1. システムが、「リンクが無効です。再度お手続きください」エラーを表示する(400)。確認状態は更新されない

**関連する概念エンティティ**: 顧客(CUSTOMER)
**関連する画面**: ログイン・新規登録画面(S-005)、プロフィール画面(S-007。未確認バッジ・再送ボタン)

**備考**:
- メールアドレスが未確認(`is_verified=false`)でも、ログイン・商品購入等を制限しない。これは実装由来で業務未確認の暫定仕様(CON-006/TBD-004)であり、プロジェクト所有者が本番化前に許可操作を決定する
- 確認用トークンの有効期限(7日間)は、パスワードリセット(UC-009、24時間)よりも長く設定した。パスワードリセットと異なりセキュリティ上の緊急性が低く、顧客がメールにすぐ気づかない可能性を考慮したため
- 確認リンクのクリックには、パスワードリセットと異なりフォーム入力を要さない(トークンのみで完結する)ため、専用の確認画面は設けず、既存のトースト通知の仕組みでフィードバックする(`03_sequence_diagram.md`参照)
