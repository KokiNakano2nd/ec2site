# UC-006: 注文をキャンセルする

[← ユースケース一覧に戻る](../01_use_cases.md)

**業務領域**: 商品購入業務

**元になったUser Story**: US-021
**主アクター**: 顧客
**ステークホルダーと関心事**:
- 顧客: 発送されてしまう前に、確実に注文を取り消し返金を受けたい
- 運営: キャンセルによって在庫・クーポン使用回数・売上集計の整合性が崩れないようにしたい。Stripeで決済済みの注文は確実に返金したい

**事前条件**: 顧客がログイン済みであり、対象の注文が当該顧客自身のものである
**事後条件(成功時)**: 対象注文のステータスが`cancelled`になる。注文明細の数量分だけ商品在庫が加算される。クーポンを使用していた場合は使用回数(`used_count`)が1減算される。Stripe決済済み(`stripe_payment_intent_id`が存在する)の場合はStripe側で返金(Refund)が実行される。顧客にステータス変更通知メールが送信される

**基本フロー**:
1. 顧客が、注文履歴画面(S-004)で対象の注文の「キャンセルする」を選択する
2. システムが、対象注文が当該顧客のものであり、ステータスが`pending`または`processing`であることを確認する
3. システムが、対象注文に`stripe_payment_intent_id`が設定されている場合、Stripe返金API(`stripe.Refund.create`)を呼び出す
4. システムが、注文明細(`order_items`)の数量分だけ、対応する商品(`products`)の在庫(`stock`)を加算する
5. システムが、対象注文にクーポンが適用されていた場合、当該クーポンの`used_count`を1減算する
6. システムが、対象注文のステータスを`cancelled`に更新する
7. システムが、ステータス変更通知メールを送信する(N-002拡張。送信失敗時もフロー自体は継続する)

**代替フロー**: なし

**例外フロー**:
- 2a. 対象注文が存在しない、または当該顧客のものでない場合
  1. システムが、「注文が見つかりません」エラーを表示する(404)
- 2b. 対象注文のステータスが`pending`/`processing`以外の場合
  1. システムが、「発送済みの注文はキャンセルできません」エラーを表示する(400)。注文は変更されない
- 3a. Stripe返金APIの呼び出しが失敗した場合
  1. システムが、「返金処理に失敗しました」エラーを表示する(500)。注文ステータス・在庫・クーポン使用回数は変更しない(返金の成否と在庫・ステータスの整合性を保つため、返金確認後にDB更新を行う)

**関連する概念エンティティ**: 顧客(CUSTOMER)、注文(ORDER)、注文明細(ORDER_ITEM)、商品(PRODUCT)、クーポン(COUPON)
**関連する画面**: 注文履歴画面(S-004)

**備考**:
- 発送済み以降のキャンセルを認めない理由: 発送後は現物が既に顧客の手元に向かっているため、単純な取り消しではなく返品(現物の返送)を伴う。そのため発送済み以降はUC-007(返品申請)で扱う
- Stripe決済を使わずに確定した注文(`stripe_payment_intent_id`が`null`)の場合、返金APIは呼び出さない(Stripeに決済記録がないため)。この場合の返金(代金引換・銀行振込分の返金等)は運用側の手動対応とし、本ユースケースのスコープ外とする
