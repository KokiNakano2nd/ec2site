# UC-007: 返品を申請する

[← ユースケース一覧に戻る](../01_use_cases.md)

**業務領域**: 商品購入業務

**元になったUser Story**: US-022
**主アクター**: 顧客
**ステークホルダーと関心事**:
- 顧客: 発送後に届いた商品に問題があった場合、返品・返金の手段を持ちたい
- 運営: 現物が実際に返送されたことを確認してから返金・在庫戻しを確定させたい(顧客の申請のみで即時返金してしまうと、現物が返送されないまま返金される不正利用のリスクがある)

**事前条件**: 顧客がログイン済みであり、対象の注文が当該顧客自身のものである
**事後条件(成功時)**: 対象注文のステータスが`return_requested`になり、入力された返品理由が保存される。この時点では在庫・クーポン使用回数・返金は変更されない。管理者に(画面上の一覧を通じて)申請が可視化される

**基本フロー**:
1. 顧客が、注文履歴画面(S-004)で対象の注文の返品理由を入力し、「返品を申請する」を選択する
2. システムが、対象注文が当該顧客のものであり、ステータスが`shipped`であることを確認する
3. システムが、対象注文のステータスを`return_requested`に更新し、入力された返品理由を保存する
4. システムが、ステータス変更通知メールを送信する(N-002拡張。「返品申請を受け付けました」。送信失敗時もフロー自体は継続する)

**代替フロー**: なし

**例外フロー**:
- 2a. 対象注文が存在しない、または当該顧客のものでない場合
  1. システムが、「注文が見つかりません」エラーを表示する(404)
- 2b. 対象注文のステータスが`shipped`以外の場合
  1. システムが、「この注文は返品を申請できません」エラーを表示する(400)。注文は変更されない

**関連する概念エンティティ**: 顧客(CUSTOMER)、注文(ORDER)
**関連する画面**: 注文履歴画面(S-004)

**備考**:
- 承認・却下の判断と、それに伴う在庫・クーポン使用回数・返金の確定はUC-008(管理者による返品承認・却下)で行う。本ユースケースは申請の受付のみを担う
- 返品理由は自由記述の任意入力とし、業務上の必須項目ではない(未入力の場合は空文字として保存する)
