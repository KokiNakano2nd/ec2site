# UC-005: 退会する

[← ユースケース一覧に戻る](../01_use_cases.md)

**業務領域**: 会員管理業務

**元になったUser Story**: US-020
**主アクター**: 顧客
**ステークホルダーと関心事**:
- 顧客: 意図せず退会してしまわないようにしたい。退会後に自分の個人情報が残り続けないようにしたい
- 運営: 退会後も売上分析・注文履歴上の整合性(合計金額・注文件数)は保ちたい。退会を装った不正操作(他人のアカウントの乗っ取り退会等)を防ぎたい

**事前条件**: 顧客がログイン済みである
**事後条件(成功時)**: アカウントが論理削除される(`is_active=false`)。メールアドレス・ハッシュ化パスワードが匿名化された値に置き換えられる。配送先(`addresses`)・お気に入り(`favorites`)は削除される。注文履歴(`orders`/`order_items`)・レビュー(`reviews`)は業務記録として削除せず残す(投稿者表示は匿名化後のメールアドレスになる)。ログアウトされ、以後そのアカウントでのログインはできなくなる

**基本フロー**:
1. 顧客が、プロフィール画面(S-007)で「退会する」を選択する
2. システムが、確認のため現在のパスワードの入力を求める
3. 顧客が、現在のパスワードを入力し、確定する
4. システムが、入力されたパスワードが現在のログインユーザーのものと一致することを確認する
5. システムが、配送先(`addresses`)・お気に入り(`favorites`)を削除する
6. システムが、`users`テーブルの当該レコードについて、メールアドレスを匿名化した値(例: `deleted-user-{id}@deleted.invalid`)に置き換え、パスワードを再利用不能なハッシュ値に置き換え、`is_active`を`false`に、`deleted_at`を現在時刻に設定する
7. システムが、退会完了メールを送信する(送信失敗時もフロー自体は継続する。N-003参照)
8. システムが、クライアント側のログイン状態を破棄し、ログイン前の画面に遷移する

**代替フロー**: なし

**例外フロー**:
- 4a. 入力されたパスワードが一致しない場合
  1. システムが、「パスワードが正しくありません」エラーを表示する(403)。アカウントは削除されない

**関連する概念エンティティ**: 顧客(CUSTOMER)、配送先(ADDRESS)、お気に入り(FAVORITE)、注文(ORDER、削除対象外)、レビュー(REVIEW、削除対象外)
**関連する画面**: プロフィール画面(S-007)

**備考**:
- 注文履歴(ORDER/ORDER_ITEM)を削除しない理由: 売上分析業務(F-029)の集計値やクーポン使用回数(`used_count`)の整合性を退会によって壊さないため。退会後もこれらは「匿名化されたユーザーによる過去の注文」として業務記録に残る
- レビュー(REVIEW)を削除しない理由: 他の顧客の購入判断の参考情報として、投稿者本人の退会後も価値を持つため。ただし投稿者のメールアドレス表示は匿名化後の値になる
- 個人情報の保持範囲・匿名化方針の詳細は[[../06_nonfunctional_requirements|06_nonfunctional_requirements.md]] NFR-013・NFR-016を参照
