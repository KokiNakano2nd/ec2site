# 画面設計

## 商品購入業務

### S-001: 商品一覧画面

**概要**: 商品を閲覧し、カートに追加する画面(`view === "products"`、`ProductList`/`ProductDetail`)
**元になった画面一覧項目**: S-001(`05_screen_list.md`)

**表示項目**:
| 項目名 | 内容 | 表示条件 |
|---|---|---|
| 商品名 | `Product.name` | 常時 |
| 商品説明 | `Product.description` | 常時(一覧では2行で省略表示) |
| 価格 | `Product.price`(税抜表示) | 常時 |
| 在庫状況 | 「在庫 {stock}」/「残りわずか」(`stock <= 7`)/「在庫切れ」(`stock === 0`) | 常時 |
| 商品画像 | `Product.image_url` | 常時 |
| 追加画像 | `GET /products/{id}/images`の結果を`display_order`順に表示 | 商品詳細表示時、登録がある場合 |
| お気に入り状態 | ❤️(登録済み)/🤍(未登録) | ログイン時のみ |
| おすすめ商品 | 同カテゴリの商品一覧(`GET /products/{id}/recommendations`) | 商品詳細表示時 |
| レビュー一覧・平均評価 | `GET /products/{id}/reviews` | 商品詳細表示時 |

**入力項目**:
| 項目名 | 型 | 必須 | バリデーション | エラー時の表示 |
|---|---|---|---|---|
| 検索キーワード | 文字列 | 任意 | なし(クライアント側で商品名・説明文に対する部分一致フィルタ。サーバーAPIへは送信しない) | 該当なしの場合「{キーワード}に一致する商品が見つかりませんでした」 |
| 購入数量(商品詳細) | 整数 | 必須(デフォルト1) | 1以上。在庫超過分は`POST /cart`側で400エラーとして検出 | 「在庫数を超える数量は指定できません」 |
| お気に入りボタン(商品詳細・カード) | ボタン(トグル) | - | ログイン時のみ操作可(未ログイン時はS-005へ遷移) | なし(F-011) |
| レビュー評価(商品詳細) | 整数(星1〜5、`StarRating`) | 必須(デフォルト5) | 1〜5の範囲(UIコンポーネント上は常に範囲内の値のみ選択可能) | 「評価は1〜5で指定してください」(サーバー側チェック) |
| レビューコメント(商品詳細) | 文字列 | 任意 | なし | - |
| レビュー投稿ボタン(商品詳細) | ボタン | - | ログイン時のみ投稿可(未ログイン時はS-005へ遷移)。同一商品への2回目の投稿はエラー | 「すでにレビューを投稿済みです」(UC-004) |

**画面遷移**:
| 操作 | 遷移先 | 条件 |
|---|---|---|
| 商品カードをクリック | S-001(商品詳細表示) | 常時 |
| ヘッダーの「カート」をクリック | S-002 カート画面 | 常時 |
| お気に入りボタン/レビュー投稿ボタン押下 | S-005 ログイン・新規登録画面 | 未ログイン時 |

### S-002: カート画面

**概要**: カート内商品の確認・数量変更・クーポン適用・注文確定を行う画面(`view === "cart"`、`CartView`)。カード決済を使わない即時注文確定(UC-003)と、Stripeによるカード決済(UC-002)の2つの確定手段を持つ
**元になった画面一覧項目**: S-002(`05_screen_list.md`)

**表示項目**:
| 項目名 | 内容 | 表示条件 |
|---|---|---|
| 商品名・単価・数量 | カート内商品(`CartItemOut`) | カートに商品がある場合 |
| 小計 | `Σ(単価 × 数量)` | 常時 |
| 割引額 | 適用中クーポンによる割引額 | クーポン適用時のみ(`discountAmount > 0`) |
| 送料 | 「無料」(固定表示) | 常時 |
| 消費税(10%) | `割引後小計 × 0.1` | 常時 |
| 合計金額(税込) | `割引後小計 + 消費税` | 常時 |
| 配送先一覧 | `GET /addresses` の結果 | 登録済み配送先がある場合 |

**入力項目**:
| 項目名 | 型 | 必須 | バリデーション | エラー時の表示 |
|---|---|---|---|---|
| 数量(カート内商品) | 整数 | 必須 | 1以上、在庫数(`stock`)以下 | 「在庫数を超える数量は指定できません」 |
| クーポンコード | 文字列 | 任意 | 有効(`is_active`)なクーポンと一致し、使用回数が上限未満であること | 「無効なクーポンコードです」/「このクーポンは使用回数の上限に達しています」 |
| 配送先選択 | 選択(ラジオボタン) | 任意 | 登録済み配送先から選択 | なし |

- **要求ギャップ**: 選択した配送先(`selectedAddressId`)は画面状態として保持されるが、`createOrder`/`createCheckoutSession`へ送られず、注文にも保存されない。画面上の選択が注文へ反映されると誤認させるため、TBD-006としてプロジェクト所有者が「注文へ配送先snapshotを保存する」か「選択UIを削除する」か決定するまで、本番化を不可とする

**画面遷移**:
| 操作 | 遷移先 | 条件 |
|---|---|---|
| 「注文を確定する」ボタン押下 | S-004 注文履歴画面 | カートが空でない、在庫が十分(UC-003) |
| 「カードで決済(Stripe)」ボタン押下 | S-003 Stripeの決済ページ(外部) | カートが空でない、Stripe決済が有効(`GET /config`の`stripe_enabled`がtrue)(UC-002) |

### S-003: Stripe決済ページ(外部)

**概要**: クレジットカード情報を入力し決済する、Stripeが提供する外部画面。自システムはこの画面のレイアウト・入力項目を管理しない
**元になった画面一覧項目**: S-003(`05_screen_list.md`)

- 画面のレイアウト・入力項目の詳細はStripe公式ドキュメントを参照(自システムの管理範囲外)。自システムとの接続点(送信データ・受信データ)は `03_external_interface.md` に記載する

**画面遷移**:
| 操作 | 遷移先 | 条件 |
|---|---|---|
| 決済成功 | S-004 注文履歴画面 | `payment_status === "paid"`(`success_url`へリダイレクト後、`POST /payment/complete`が成功) |
| 決済キャンセル | S-002 カート画面 | 顧客がStripe決済ページでキャンセルを選択(`cancel_url`へリダイレクト) |

### S-004: 注文履歴画面

**概要**: 確定済みの注文結果を確認する画面(`view === "orders"`、`OrderHistoryView`)。発送前の注文はキャンセル、発送済みの注文は返品申請を行える(2026-07-11更新)
**元になった画面一覧項目**: S-004(`05_screen_list.md`)

**表示項目**:
| 項目名 | 内容 | 表示条件 |
|---|---|---|
| 注文日時 | `Order.created_at` | 常時 |
| 注文番号 | `Order.id` | 常時 |
| ステータス | `Order.status`(`pending`/`processing`/`shipped`/`cancelled`/`return_requested`/`returned`等) | 常時 |
| 合計金額(税込) | `Order.total_price` | 常時 |
| 注文明細(商品名・数量・小計) | `Order.items`(`OrderItemOut`) | 常時 |
| 返品理由 | `Order.return_reason` | ステータスが`return_requested`/`returned`かつ理由が入力されている場合 |

**入力項目**:
| 項目名 | 型 | 必須 | バリデーション | エラー時の表示 |
|---|---|---|---|---|
| 「キャンセルする」ボタン(行ごと) | ボタン | - | ステータスが`pending`/`processing`であること | 「発送済みの注文はキャンセルできません」 |
| 返品理由入力欄+「返品を申請する」ボタン(行ごと) | テキスト(任意)+ボタン | - | ステータスが`shipped`であること | 「この注文は返品を申請できません」 |

**画面遷移**: なし(本画面が経路の終点)

**概要**: メールアドレス・パスワードでログイン、または新規登録する画面(`view === "login"` / `"register"`、`AuthView`)。ログイン・新規登録は同一コンポーネントでモード切替する

**表示項目**:
| 項目名 | 内容 | 表示条件 |
|---|---|---|
| モード見出し | 「ログイン」/「アカウントを作成」 | 常時 |
| モード切替リンク | 「新規登録」(ログイン時)/「ログインへ」(新規登録時) | 常時 |
| パスワードをお忘れですか?リンク | S-008(パスワードリセット要求画面)へ遷移するリンク(2026-07-13追加、US-026) | ログインモード時のみ |
| エラーバナー | サーバーから返されたエラーメッセージ | エラー発生時 |

**入力項目**:
| 項目名 | 型 | 必須 | バリデーション | エラー時の表示 |
|---|---|---|---|---|
| メールアドレス | 文字列(email) | 必須 | HTML5の`type="email"`による形式チェックのみ(サーバー側の重複チェックはF-009) | 「このメールアドレスは既に登録されています」(新規登録時) |

**画面遷移**:
| 操作 | 遷移先 | 条件 |
|---|---|---|
| ログイン/新規登録成功 | S-001 商品一覧画面 | 常時 |
| 「パスワードをお忘れですか?」クリック | S-008 パスワードリセット要求画面 | 常時(2026-07-13追加) |

**表示項目**:

|---|---|---|
| アカウント情報 | メールアドレス、「管理者」/「一般ユーザー」バッジ | 常時 |
| メール未確認バッジ(2026-07-13追加、US-027) | 「メールアドレスが未確認です」バッジと「確認メールを再送する」ボタン | `is_verified === false`の場合のみ |

| 空状態 | 配送先が0件の場合は一覧が空(専用の空状態メッセージは実装上確認できていない) | 配送先が0件の場合 |

| 項目名 | 型 | 必須 | バリデーション | エラー時の表示 |
|---|---|---|---|---|
| お名前(宛名) | 文字列 | 必須 | HTML5の`required`のみ | - |
| 電話番号 | 文字列 | 任意 | なし | - |
| 郵便番号 | 文字列 | 必須 | HTML5の`required`のみ(形式チェックはサーバー・フロントエンドいずれにも実装なし) | - |
| 都道府県 | 文字列 | 必須 | HTML5の`required`のみ | - |
| 市区町村 | 文字列 | 必須 | HTML5の`required`のみ | - |
| 番地・建物名 | 文字列 | 必須 | HTML5の`required`のみ | - |
| 部屋番号等 | 文字列 | 任意 | なし | - |
| デフォルト配送先に設定する | チェックボックス | 任意 | なし(チェック時、他の配送先のデフォルト指定を解除) | - |

- **実装上の注記**: `frontend/src/pages/ProfileView.jsx`(`ProfileView`)を確認したところ、配送先の「編集」機能はUIから提供されておらず、登録・削除・デフォルト設定のみが可能。バックエンド(`PATCH /addresses/{id}`)には編集APIが実装されているが、フロントエンドから呼び出されていない(`03_function_list.md` F-019参照)

**退会(2026-07-11追加、F-030・UC-005)**:
| 項目名 | 内容 |
|---|---|
| 「退会する」ボタン | 画面下部の「危険な操作」セクションに配置。押下すると確認モーダルを表示する |
| 確認モーダル | 現在のパスワードの入力欄と「退会を実行する」ボタン。パスワード不一致時は`DELETE /users/me`の403エラーメッセージをモーダル内に表示する |
| 実行後 | 成功時はログアウト処理を行い、ログイン画面(S-005)に遷移する |

**画面遷移**: 退会実行成功時のみ、S-005(ログイン・新規登録画面)へ遷移する。それ以外は本画面内で完結する

**メールアドレスの本人確認(2026-07-13追加、F-037・UC-010)**:
| 項目名 | 内容 |
|---|---|
| 「確認メールを再送する」ボタン | `is_verified === false`の場合のみ表示。押下すると`POST /auth/verify-email/resend`を呼び出し、成功時はトースト通知で「確認メールを再送しました」を表示する |

- 確認リンク(`?verify_token=...`)を開いた際の結果表示(成功/失敗)は、S-007ではなくアプリ全体で共通のトースト通知の仕組みを使う(`MainView`がアプリ起動時にクエリパラメータを検知して自動的に`POST /auth/verify-email/confirm`を呼び出すため、特定の画面に紐づかない。`03_sequence_diagram.md`参照)
