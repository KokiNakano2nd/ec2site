# テスト計画

## 1. 目的

機能要求と非機能要件が設計どおりに実装され、既知のリスクが許容水準まで低減されたことを、再現可能な証拠で示す。本書はテスト戦略と完了基準を定義し、個別対応は[要求トレーサビリティマトリクス](01_traceability_matrix.md)を正本とする。

## 2. 対象と対象外

### 対象

- backendの認証・認可、商品、カート、クーポン、注文、決済、返金、配送先、管理者、通知
- frontendの主要画面、状態表示、入力・エラー、画面間遷移
- API契約、DBトランザクション、外部サービス境界
- CI品質ゲート、性能、セキュリティ、アクセシビリティ、バックアップ復元

### 現段階の対象外

- 実カード課金、実顧客へのメール送信
- 未定義のproduction環境に対するSLA判定
- 配送業者、会計等、システム要求でスコープ外とした外部連携

対象外は本番化時に再評価する。Stripe sandbox契約試験は実課金とは分離して計画対象とする。

## 4. 環境とテストデータ

| 環境 | DB | Stripe/SMTP | 用途 |
|---|---|---|---|
| backend test | `TemporaryDirectory`内SQLite、各テストでdrop/create | monkeypatchまたは無効 | API/統合 |
| frontend component | jsdom、DBなし | APIを`vi.mock` | UI単体 |
| E2E | Playwright用一時SQLite | 無効 | ブラウザ結合 |
| CI | GitHub-hosted runner | 実外部呼出しなし | 回帰・品質ゲート |
| staging | 未定 | Stripe test mode、テスト用SMTPを予定 | 本番相当前検証 |

- テスト用ユーザー、注文、メールは架空データだけを使用する
- production DBを自動テストから参照しない。接続先を起動時に検査する
- Stripeキーを使う試験はtest modeを確認し、課金上限・実行責任者を定める
- 並び順や時刻境界の試験では現在時刻を固定し、ランダム推薦はseedまたは候補集合で判定する

## 5. リスクベース優先順位

| 優先度 | 対象 | 必須シナリオ |
|---|---|---|
| P0 | 決済・返金・注文確定 | 二重送信、途中失敗、再試行、権限、在庫/クーポン整合性 |
| P0 | 認証・認可・個人情報 | 他者リソース、管理者境界、退会、トークン期限・再利用 |
| P1 | カート・クーポン・在庫 | 境界値、同時更新、空状態、不正入力 |
| P1 | API契約・DB移行 | 後方互換性、失敗時復旧、既存データ保持 |
| P2 | 表示・分析・推薦 | 空状態、並び順、大量データ、ブラウザ差 |

## 6. 開始・終了基準

### 開始基準

- 対象要求/受け入れ条件と設計がレビュー可能な状態
- テスト環境がproductionから隔離されている
- 必要なfixture/mockと期待結果が定義済み
- 外部サービス試験ではtest/sandbox資格情報が確認済み

### Pull Request終了基準

- 変更対象の受け入れ条件に正常・異常・認可テストがある
- backend/frontendの該当CIが全成功
- OpenAPI差分が意図どおりで、追跡表が更新済み
- 新規P0/P1欠陥が0件。既知欠陥は理由・回避策・期限を記録

### 本番リリース終了基準

- P0機能がすべて`直接検証`
- NFR本番化ゲートが完了し、性能・アクセシビリティ・復元・移行の証拠がある
- stagingでStripe test mode、SMTP、DB migration、rollback/roll-forwardを確認
- 未解決Sev-1/Sev-2が0件

## 7. 合否と欠陥管理

| 重大度 | 定義 | リリース判断 |
|---|---|---|
| Sev-1 | データ消失、二重課金、重大な情報漏えい、全停止 | 修正・再検証まで不可 |
| Sev-2 | 主要フロー不能、認可不備、金額/在庫不整合 | 原則不可。例外は所有者承認と期限必須 |
| Sev-3 | 代替手段のある機能不具合、限定的表示崩れ | リスク評価して判断 |
| Sev-4 | 文言、軽微なUI、改善提案 | バックログ化可 |

欠陥には再現手順、期待/実際、環境、要求ID、証拠、重大度、原因、修正コミット、再検証結果を記録する。
