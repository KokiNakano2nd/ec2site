# 業務フロー図: 商品購入業務

[← 業務フロー図一覧に戻る](../01_business_flow.md) / 全体ルール: [[../../../README|docs/README.md]]

### 概要

顧客が商品をカートに追加し、クーポンを適用した上で注文を確定、決済を完了するまでの業務。ECサイトの中核となる業務フロー。

### 登場アクター

- 顧客
- システム(EC_SITE)
- Stripe(決済代行サービス)

### 業務フロー図(As-Is)

現状(システム未導入)を想定した、電話・カタログ注文ベースの業務。

```mermaid
flowchart TD
    subgraph 顧客
        A((●)) --> B[カタログから商品を選ぶ]
        B --> C[電話またはFAXで注文する]
    end
    subgraph 受注担当者
        D[注文内容を聞き取る]
        E{在庫があるか}
        F[クーポン適用の可否を確認する]
        G[請求金額を計算する]
        H[代金引換または銀行振込を案内する]
    end
    C --> D
    D --> E
    E -->|なし| I[欠品を伝え注文を断る]
    E -->|あり| F
    F --> G
    G --> H
    H --> J((◎))
    I --> J
```

### 課題・問題点

- 在庫確認や金額計算が手作業のため、計算ミスや在庫の二重販売が発生しやすい
- 電話・FAXでの受付は営業時間内に限られ、顧客が注文したいタイミングで注文できない
- クーポン適用可否の判断が受注担当者の目視確認に依存しており、不正利用のチェックが難しい

### To-Beにおける主な変更点

- 在庫確認・金額計算(小計・クーポン割引・税)・決済処理をすべてシステム化し、24時間いつでも注文可能にする
- クーポンの有効性判定(有効期限・利用回数上限)をシステムが自動判定し、不正利用のリスクを下げる
- 決済方法として「Stripeによるカード決済」と「カード決済を使わない即時確定(代金引換・銀行振込等を想定)」の2経路を用意し、決済はカード決済を選んだ場合のみStripeに委任する(PCI DSS対応の簡略化)。この2経路の分岐は、外部設計フェーズで`frontend/src/pages/CartView.jsx`(`CartView`)を確認した結果判明したため、業務フロー図(本ドキュメント)に遡って追記した(該当ユースケース: UC-002, UC-003)

### 業務フロー図(To-Be): 注文キャンセル・返品申請(2026-07-11追加)

以前は、注文確定後に顧客側から注文を取り消す手段が存在しなかった(管理者がステータスを更新する以外に注文状態を変える手段がない)ことが機能一覧上のギャップとして認識されていたため、新機能として追加する。発送前後で扱いを分け、発送前は顧客の即時判断でキャンセル、発送後は現物のやり取りを伴うため管理者承認を要する「返品申請」とする。

```mermaid
flowchart TD
    subgraph 顧客
        A((●)) --> B["注文履歴画面で対象の注文を選ぶ<br/><<システム化>>"]
        B --> C{"注文のステータスは?<br/><<システム化>>"}
        C -->|受付中/処理中| D["「注文をキャンセルする」を選択する<br/><<システム化>>"]
        C -->|発送済み| E["返品理由を入力し「返品を申請する」を選択する<br/><<システム化>>"]
    end
    subgraph システム
        F["Stripe決済済みの場合は返金処理を行う<br/><<システム化>>"]
        G["在庫を注文数量分戻し、クーポン使用回数を戻す<br/><<システム化>>"]
        H["注文ステータスをキャンセル済みにする<br/><<システム化>>"]
        I["注文ステータスを返品申請中にする<br/><<システム化>>"]
        J["ステータス変更通知メールを送信する<br/><<システム化>>"]
    end
    D --> F
    F --> G
    G --> H
    H --> J
    E --> I
    I --> J
    J --> K((◎))
```

- 返品申請(発送済み以降)は、この時点では在庫・クーポン使用回数を戻さず、注文管理業務(管理者向け、`08_order_admin.md`)側の承認処理で戻す(現物が実際に返送されたことを管理者が確認してから在庫・返金を確定させるため)
- 該当ユースケース: UC-006(キャンセル)、UC-007(返品申請)
